# Workflow — clarify-question

활성화 조건은 `agent/skills/clarify-question/SKILL.md` 참조. 이 문서는 활성화 후 절차만 다룬다.

---

## Step 1: 모호점 감지

질문을 읽고 답이 크게 갈릴 만한 **2-3가지 핵심 모호점**을 식별한다. 사소한 표현상 차이는 건너뛴다.

스캔 항목:
- 지시 대상이 명확한가? (이거/저거/그것)
- 용어가 단일 해석인가? (예: "최적화" = 컴파일러 vs 성능 vs 수학)
- 범위가 정해졌는가? (개요 vs 전체 vs 특정 시나리오)
- 깊이가 정해졌는가? (1줄 vs 개념 vs 수학적 상세)

## Step 2: 짧게 짚고 기본 진행 선언

답변 **시작 부분에** 간단히 제시:

```
"X"는 [N가지 해석]이 가능합니다. **기본적으로 [A 해석]으로 답하겠습니다.**
다른 방향이면 알려주세요:
- A: [가장 가능성 높은 해석]
- B: [대안 해석]
- (C: [필요시])
```

규칙:
- 후보는 **최대 3개**.
- 후보가 1개로 명확하면 이 단계 생략하고 바로 본문으로.
- 사용자 질문보다 길게 쓰지 않는다.

## Step 3: 즉시 본문 답변

추가 입력 기다리지 않고 **그대로 답변 진행**. 기본 해석(A)으로 본문 작성.

## Step 4: 사용자 반응 처리

- "B 방향이었어" → 즉시 B로 재답변 (기존 A 답변 폐기 명시)
- "그대로 진행" / 무반응 → 이후 같은 사용자 세션에서 이 스킬을 덜 공격적으로 적용
- "추가 질문 그만" → 이 세션에서 스킬 비활성화

---

## 절대 하지 말 것

- ❌ "질문을 명확히 해주세요"라며 답변 보류
- ❌ 4개 이상의 후보 나열
- ❌ 컨텍스트로 이미 명확한 것을 다시 묻기
- ❌ 사용자 질문을 비판하는 뉘앙스 ("질문이 불명확하다")
- ❌ 사소한 표현상 모호점까지 나열

## 좋은 예

### 예 1: 범위 모호

**사용자**: "최적화에 대해 설명해줘"

```
"최적화"는 맥락에 따라 답이 크게 달라집니다. 기본적으로 **컴파일러 최적화 개요**로
답하겠습니다. 다른 방향이면 알려주세요:
- A: 컴파일러 최적화 (상수 폴딩, CSE, 루프 등)
- B: 프로그램 성능 최적화 (프로파일링, 병목 제거)
- C: 수학적 최적화 (ILP, 경사하강법)

[A 본문 진행...]
```

### 예 2: 지시 대상 불명

**사용자**: "이거 왜 이렇게 동작해?"

```
"이거"는 직전 대화의 [구체 주제]로 이해하고 답합니다. 다른 것이면 알려주세요.

[본문 진행...]
```

### 예 3: 깊이 불명

**사용자**: "polyhedral model 설명해줘"

```
깊이에 따라 답이 갈립니다. **개념 + 수식 이해 수준**으로 답합니다. 다른 방향이면:
- 개요 (1분 요약)
- 개념 + 수식 (현재 기본)
- 수학적 상세 (증명, Farkas Lemma 등)

[본문 진행...]
```

## 나쁜 예

### 강요 (금지)

**사용자**: "polyhedral 설명"

❌ 잘못된 적용:
```
답하기 전에 알려주세요:
1. 깊이 수준?
2. 어떤 언어?
3. 예시 필요?
4. 다이어그램 원하시나요?
```

이유: 답을 보류하고 되묻기만 함.

### 과잉 분해 (금지)

**사용자** (이전 대화 컨텍스트: compiler/polyhedral): "affine 변환이란?"

❌ 잘못된 적용:
```
"affine"의 맥락이 여러 가지입니다:
- 수학의 affine 변환
- 컴파일러의 affine function
- 기하학의 affine 공간
- MLIR의 affine dialect
- polyhedral의 affine 표현식
```

이유: 컨텍스트상 명확한데 과도 분해.
