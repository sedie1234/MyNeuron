# clarify-question — 지식 질문 명확화 제안 스킬

사용자가 **지식을 얻거나 개념을 이해하려는 대화**에서, 질문이 모호할 때 방향을 제안한다. 제안만 할 뿐 **강요하지 않는다**. 사용자가 그대로 진행을 원하면 즉시 최선의 해석으로 답변한다.

---

## 활성화 조건 (언제 작동하는가)

사용자가 **학습/탐구 모드**일 때:
- "~이란 무엇인가", "~에 대해 설명해줘", "~의 차이는?"
- "왜 ~인가", "어떻게 ~인가"
- "~의 예시", "~의 배경"
- 기존 지식에 대한 질문 (예: "이 문서에서 말하는 ~는?")
- 새 개념, 이론, 원리에 대한 질문
- 논문/PDF/강의 내용에 대한 질문
- 특정 페이지/섹션에 대한 질문

**그리고** 다음 중 하나라도 해당할 때:
- 지시 대명사(이거, 저거, 그것, 여기)의 지시 대상 불분명
- 여러 해석이 가능한 단어/개념
- 범위가 넓어 답이 크게 갈림
- 컨텍스트 누락 (어떤 문서/주제/페이지인지 불명)
- 깊이 수준 불명확 (개요 vs 상세, 초보 vs 전문)

---

## 비활성화 조건 (언제 작동하지 않는가)

다음 경우에는 **스킬을 적용하지 않음**:

- **실행/수정 요청**: "이 파일 고쳐줘", "커밋해줘", "push 해줘", "파일 이동"
- **간단한 사실 질문**: "현재 디렉토리 뭐야?", "이 파일 있어?"
- **연속 대화**: 직전 답변에 이어지는 자연스러운 후속 질문 (컨텍스트가 이미 명확)
- **사용자가 명시적으로 거부**: "바로 답해줘", "추가 질문 없이", "그냥 최선으로 답해"

---

## 작동 방식

### 1단계: 모호점 감지
질문을 읽고 답이 크게 갈릴 만한 2-3가지 핵심 모호점을 식별한다. **사소한 것은 건너뛴다**.

### 2단계: 짧게 짚고 기본 진행
답변 **시작 부분에** 간단히:

> 몇 가지 해석이 가능합니다. **기본적으로 [A 해석]으로 답하겠습니다**.
> 다른 방향이면 알려주세요:
> - A: [가장 가능성 높은 해석]
> - B: [대안 해석]
> - (C: [필요시 추가])

### 3단계: 즉시 본문 답변
짚은 직후 **그대로 답변 진행**. 사용자의 추가 답을 기다리지 않는다.

### 4단계: 사용자 반응 처리
- 사용자가 다른 방향이라고 하면: 재답변
- 사용자가 그냥 진행하라고 하면: 이후 세션 동안 이 스킬을 덜 공격적으로 사용

---

## 절대 하지 말 것

### 강요
- ❌ "질문을 명확히 해주세요"
- ❌ "답하기 전에 알려주세요: ..."
- ❌ 답변을 보류하고 되묻기만 하기

### 과도한 분해
- ❌ 사소한 표현상 모호점까지 다 나열
- ❌ 이미 컨텍스트로 명확한 것을 다시 묻기
- ❌ 질문보다 긴 제안

### 스타일
- ❌ "혹시 ~하셨나요?" 같은 추측성 재질문 반복
- ❌ 사용자 질문을 비판하는 뉘앙스 ("질문이 불명확하다")

## 좋은 예시

### 예 1: 범위 모호

**사용자**: "최적화에 대해 설명해줘"

**스킬 적용**:
```
"최적화"는 맥락에 따라 답이 크게 달라집니다. 기본적으로 **컴파일러 최적화 개요**
로 답하겠습니다. 다른 방향이면 알려주세요:
- A: 컴파일러 최적화 (상수 폴딩, CSE, 루프 등)
- B: 프로그램 성능 최적화 (프로파일링, 병목 제거)
- C: 수학적 최적화 (ILP, 경사하강법 등)

[A 방향의 답변 이어서 진행...]
```

### 예 2: 지시 대상 불명

**사용자**: "이거 왜 이렇게 동작해?"

**스킬 적용**:
```
"이거"가 무엇을 가리키는지 애매합니다. 직전 대화의 [구체적 주제]로 이해하고
답하겠습니다. 다른 것이면 알려주세요.

[답변 진행...]
```

### 예 3: 깊이 수준 불명

**사용자**: "polyhedral model 설명해줘"

**스킬 적용**:
```
깊이 수준에 따라 답이 달라집니다. 기본적으로 **핵심 개념 + 수식 이해 수준**
으로 답하겠습니다. 다른 방향이면 알려주세요:
- 개요 (1분 요약)
- 개념 수준 (현재 기본)
- 수학적 상세 (증명, Farkas Lemma 등)

[기본 수준 답변 진행...]
```

## 스킬이 피해야 할 나쁜 예시

### 나쁜 예 1: 강요

**사용자**: "polyhedral 설명"

**잘못된 스킬 적용**:
```
답하기 전에 다음을 알려주세요:
1. 깊이 수준은? (초급/중급/고급)
2. 어떤 언어로?
3. 예시가 필요한가요?
4. 다이어그램을 원하시나요?
```

→ ❌ 강요, 답을 보류함

### 나쁜 예 2: 과잉 분해

**사용자**: "affine 변환이란?"

**잘못된 스킬 적용**:
```
"affine"의 맥락이 여러 가지입니다:
- 수학의 affine 변환
- 컴파일러의 affine function
- 기하학의 affine 공간
- MLIR의 affine dialect
- polyhedral의 affine 표현식
- ...
```

→ ❌ 컨텍스트상 명확한데 과도하게 분해 (이전 대화가 컴파일러/polyhedral이면 바로 답하면 됨)

---

## 세션 중 사용 패턴

1. **초반**: 사용자 성향 파악 전이므로 적극 적용
2. **중반**: 사용자가 반복적으로 "바로 답해달라"고 하면 덜 적용
3. **사용자 스타일 학습**: 사용자가 이미 상세한 질문을 던지는 스타일이면 거의 적용 안 함

---

## 스킬의 경계

### 이 스킬은 "제안"
- 답변 진행을 막지 않는다
- 사용자의 의지를 최우선으로 한다
- 모호해도 최선의 해석으로 답변 제공

### 이 스킬이 아닌 것
- "애매한 판단 금지" 원칙과 다름: 저 원칙은 **중요한 행동 결정**(파일 삭제, 카테고리 결정 등)에 적용, 이 스킬은 **질문 해석**에만 적용
- "되묻기"가 아님: 짧게 짚고 즉시 진행
