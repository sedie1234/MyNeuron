# 지시
- 지식 축적용 agent 생성
- agent의 내용은 agent 디렉토리 내에 생성하고, agent 실행은 _MyNeuron/에서 실행할 것이므로 _MyNeuron/에는 claude code cli 실행 시, agent 디렉토리에서 설정을 자동으로 가져올 수 있도록 하는 설정만 남긴다. 

## 실험지식용 agent 생성
- 실험에 대한 내용을 지식으로 전이하는 것을 도와주는 agent 생성
- 이 agent의 목적은 실험 내용을 파악하고, 성공 및 실패한 내용과 더불어 이로부터 얻은 지식을 축적할 수 있는 형태의 지식으로 변환해주는 것이 목적이다.
- 이 지식 축적용 디렉토리에 추가할 수 있도록 문서를 제작하는 것이 주요 역할이다.
- 사용자는 이 agent를 복사하여 본인의 프로젝트에서 이 agent를 활용할 수 있고, agent는 실험에 대한 개요, 실험 계획 수립, 실험 진행, 실험 결과, 실험 과정이나 실험 결과로 부터 얻은 지식과 경험을 이 agent에게 전달할 수 있는 형태로 정리하여 준다.

## 프로젝트 지식용 agent 생성
- 특정 프로젝트를 진행하며 얻은 지식을 업데이트 하는 agent 생성
- 실험 지식용 agent의 상위호환이다. 
- 차이점이라면 프로젝트의 개요와 진행사항에 대해서 파악할 수 있어야 한다.
- 매우중요 : 프로젝트에는 개인 또는 회사만의 지식재산이 있을 수 있다. 이 지식재산은 축적할 지식에 포함하지 않도록 주의한다. 
- 지식재산은 사용자가 허락한 경우에 한해서만 축적할 지식으로 전환할 수 있다.
- 지식재산 여부가 확실하지 않은 경우, agent는 반드시 사용자에게 질문하여 확인한 후 처리한다. (이 규칙은 프로젝트 지식용 agent에만 적용된다.)

# agent의 역할

## 지식 축적
- agent는 지식을 축적한다. 내가 얻은 지식, 이론, 실험에 대한 실패 및 성공의 경험과 이로부터 얻은 것에 대한 정보들, 읽은 논문이 이 범위에 해당하며 그 범위는 더 넓어질 수 있다. 범위가 넓어지는 경우에는 agent 설정에 추가할 수 있도록 한다. 
- 지식은 기본적으로 obsidian을 활용하여 obsidian 형식의 문서로 축적을 한다.
- 지식은 시간에 따라 새로 개발되거나 새로운 사실이 발견되며 변화한다. 그러므로 지식을 얻은 시간을 기록한다.
- 지식이 업데이트 될 때, 과거의 지식은 특별한 지시가 있지 않은 이상 그대로 남겨둔다.
- 새로운 지식을 쌓으려고 할 때, 기존지식과 중복되는 부분이 있다면 병합은 하되 내용을 최소한으로 수정한다. 이것은 기존의 지식과 새로 얻은 지식이 완전히 동일하다고 판단될 때 적용된다.

## temp 문서 처리
- 사용자가 외부에서 생성한 문서를 __MyNeuron/temp/에 옮겨놓고, agent에게 정리를 지시하면 agent는 다음을 수행한다:
  - temp/ 내 문서의 내용을 분석한다.
  - 적절한 카테고리를 제안한다.
  - 사용자 승인 후, neuron/ 하위의 적절한 위치로 이동 및 정리한다.
  - 필요한 경우 내용을 분할하거나 합쳐서 적절한 형태로 지식을 축적한다.
  - 처리 완료된 문서는 temp/에서 삭제한다.

## 기억을 떠올림
- agent는 어떤 기억에 대해 질문했을 때, 관련된 지식들과 경험들을 설명할 수 있어야 한다. 
- 중복된 지식이 여러 개 있는 경우, 가장 최신의 정보를 제공한다.
- 요청이 있는 경우 지식이 어떻게 변화했는지 history를 제공한다.

## 확장 질문
- 기존 지식에 없는 지식으로도 대답에 응할 수는 있다. 하지만 이 경우에는 기존 지식에는 없는 내용임을 명시해야 한다. 

## 문서 정리
- 문서들의 내용이 충돌하는 경우가 발생할 수 있다.
- 사용자의 요청에 따라 (또는 너무 오랜 기간 문서정리를 안했다고 판단되는 경우에는 사용자에게 제안할 수 있음. 예를 들어 마지막 문서정리로부터 50개 문서가 추가되거나 한 달이 지난 경우 등을 판단하여 문서 정리를 제안할 수 있다.) 문설르 정리힌다.
- 규칙 1. 동일한 지식인 문서들이 여러 개 존재하는 경우 문서들을 병합하여 하나의 지식으로 만든다. 
- 규칙 2. 프로젝트나 실험에 있는 지식들이 모두 정리가 되었다면 실험경과와 결론, 얻은 지식을 요약하고, 이로부터 얻은 지식은 지식문서로 전이한다.
- 규칙 3. 같은 내용에 대해 상반되는 내용이 있어 충돌하는 경우, 사용자가 판단할 수 있도록 지식이 생긴 경과와 지식의 내용, 지식이 만들어진 시기를 포함하여 자세한 내용을 제공하도록 하고 사용자에 지시에 따라 문서를 정리한다.
- 필요에 따라 카테고리들을 재정리할 수도 있다.

# 디렉토리 
- 모든 지식 카테고리는 neuron/ 디렉토리 하위에 위치한다.
- 각 디렉토리의 아래에는 어떤 내용에 대한 것인지 카테고리를 다시 생성한다. 
- 이후에 직접 지식을 찾거나, rag를 통한 지식 탐색 때, 효율적으로 찾는 것을 목적으로 한다. 
- 정해진 카테고리 말고도 유용한 카테고리가 있다면 제안하고 추가하는 형식으로 유지보수 할 수 있어야 한다.
- 향후 카테고리가 어지럽혀져서 정리하는 것이 좋다고 판단되는 경우에는 제안을 주고 정리하여 지식을 유지보수 할 수 있어야 한다.
- 목적에 맞는 적절한 tree를 형성하여야 한다.

## 디렉토리 구조
```
__MyNeuron/
├── agent/           # agent 설정 파일
├── temp/            # 외부 문서 임시 저장소
├── tools/           # 이미지 생성 등 각종 도구
├── order/           # 지시서
└── neuron/          # 지식 저장소 루트
    ├── 이론/
    ├── 실험/
    ├── 프로젝트/
    ├── 대화/
    ├── 지식/
    └── images/      # 모든 이미지 통합 저장
```

## 이론 카테고리
- 이론에 관한 지식들을 축적한다.
- 이론에 대해 의문을 제기할 수는 있지만 이 지식들은 영원히 축적하는 것을 목적으로 한다.

## 실험에 대한 실패 및 성공의 경험
- 나는 궁금증이나 목적에 따라 실험을 계획하고, 실제 실험을 진행한다.
- 이 실험에 대해 성공과 실패를 기록하고, 얻은 정보들을 축적한다.
- 실험 카테고리의 아래에는 목적에 맞게 다시 적절한 카테고리들로 구축한다.
- 외부의 실험에서 얻은 지식문서는 사용자가 직접 __MyNeuron/temp/에 옮기고, 사용자의 정리 지시 시 agent가 적절한 카테고리에 지식을 축적한다.
- 필요한 경우, 내용을 분할하거나 합쳐서 적절한 형태로 지식을 축적하기도 한다.
- 실험에서 얻은 확고한 지식은 지식 카테고리로 전이할 수 있다. 

## 프로젝트로 인해 발생한 지식
- 프로젝트를 제작하며 발생한 지식들을 축적한다.
- 프로젝트를 제작하며 발생한 지식은 주로 외부에서 문서가 생성되어 __MyNeuron/temp/에 옮겨지고, 사용자의 정리 지시 시 agent가 정리한다.
- 하위에 적절한 카테고리들을 생성할 수 있다.

## 대화로부터 얻은 지식
- agent와 대화를 하며 지식을 얻는 경우도 있다. 적절한 대화의 분기에 따라 대화 내용을 지식문서로 만든다. 
- 사용자의 요청이 있는 경우 대화에 대해 문서를 만든다. 
- 적절한 대화의 분기는 대화의 주제에 따라 (예를 들어 "컴파일러에 대한 대화"나 "양자 역학에 대한 대화" 등이 있을 수 있다.) 기점을 분기할 수 있다.
- 적절한 대화의 분기는 시간에 따라 (예를 들어 "방금 나눈 대화"나 "오늘 나눈 대화" 등이 있을 수 있다.) 기점을 분기할 수 있다. 
- 사용자의 요청이 아니라 직접 분기하는 경우에는 사용자에게 나눈대화를 지식으로 축적하는 것을 제안하고, 승락하는 경우에만 축적하도록 한다. 

## 지식 카테고리
- 실험, 프로젝트, 대화 또는 사용자가 직접 지식을 가져오는 경우에 이 카테고리 하위에 지식을 축적한다. 
- 하위 카테고리는 적절하게 분배하고, 문서정리 시간에 그 카테고리들이 변경될 수도 있다. 

# Frontmatter 명세
- 모든 지식 문서는 YAML frontmatter를 포함한다.
- 공통 필드와 유형별 선택 필드로 구성된다.
- frontmatter의 필드명과 태그는 영어로 통일한다.

## 공통 필드 (모든 문서에 적용)
```yaml
---
title: "문서 제목"
type: theory | experiment | project | conversation | knowledge
status: draft | confirmed | deprecated | conflicted
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [영어 태그]
related: ["[[관련문서1]]", "[[관련문서2]]"]
summary: "한 줄 요약 — RAG 검색 시 빠른 판별용"
---
```

### 필드 설명
- title: 문서 제목
- type: 문서 유형 (어떤 카테고리에 속하는지)
- status: 문서/지식의 검증 상태
  - draft: 초기 기록, 검증 전. 새 문서 생성 시 기본값
  - confirmed: 검증 완료, 신뢰할 수 있는 지식. 사용자가 확인하거나 실험으로 검증 시 전환
  - deprecated: 더 이상 유효하지 않음 (문서는 보존). 새로운 지식이 이를 대체할 때 전환
  - conflicted: 다른 지식과 모순, 사용자 판단 필요. 문서 정리 시 충돌 발견 시 전환
- created: 최초 생성일
- updated: 마지막 수정일
- tags: 검색/분류용 태그 (기술 용어는 영어)
- related: Obsidian 위키링크로 연관 문서 연결
- summary: 한 줄 요약

## 유형별 선택 필드

### experiment (실험)
```yaml
objective: "실험 목적"
hypothesis: "가설"
result: success | failure | partial | ongoing
lessons: "핵심 교훈 한 줄 요약"
transferred_to: ["[[전이된 지식문서]]"]
```
- result는 실험 자체의 성패, status는 문서/지식의 검증 상태. 예: 실패한 실험(result: failure)도 확인된 지식(status: confirmed)이 될 수 있다.

### project (프로젝트)
```yaml
project_name: "프로젝트명"
ip_cleared: true | false
phase: planning | in-progress | completed | archived
```
- ip_cleared: false인 문서는 지식 전이 시 반드시 사용자 확인을 거친다.

### conversation (대화)
```yaml
topic: "대화 주제"
conversation_date: YYYY-MM-DD
source: "Claude Code" | "동료" | "세미나" | etc.
```

### theory (이론)
```yaml
domain: "분야 (예: quantum-mechanics, compiler)"
references: ["논문/교재 출처"]
confidence: high | medium | low
```
- confidence: 이론의 확신도. 교과서 수준이면 high, 최신 연구/논쟁 중이면 low.

### knowledge (지식 — 다른 카테고리에서 전이된 확정 지식)
```yaml
origin: ["[[출처 문서]]"]
domain: "분야"
```

# 지식 연결 (Link) 전략
- Obsidian의 `[[위키링크]]` 를 적극 활용하여 지식 간 연결을 구축한다.
- 새 문서 생성 시, 기존 문서 중 관련된 문서를 탐색하여 양방향 링크를 삽입한다.
  - 새 문서의 frontmatter `related` 필드에 관련 문서 링크를 추가한다.
  - 기존 관련 문서의 `related` 필드에도 새 문서 링크를 추가한다.
- 문서 본문 내에서 다른 지식 문서에서 다루는 개념이 언급될 때, 해당 개념에 `[[위키링크]]`를 걸어 Obsidian graph view에서 연결이 보이도록 한다.
- 지식이 전이될 때 (예: 실험 → 지식), 원본 문서와 전이된 문서 간에 `origin`/`transferred_to` 필드로 출처 관계를 명시한다.
- 태그(`tags`)는 카테고리 횡단 검색을 위해 사용한다. 예를 들어 이론 문서와 실험 문서가 같은 주제를 다루면 동일한 태그를 부여하여 태그 기반 탐색이 가능하도록 한다.

# 언어 정책
- 지식 문서의 기본 작성 언어는 한국어로 한다.
- 기술 용어, 전문 용어는 영어로 표기한다. (예: "transformer", "backpropagation", "gradient descent")
- 영어 외의 원문(예: 일본어, 독일어 등)으로 표기하는 것이 더 적절하다고 판단되는 경우, agent는 사용자에게 원문 표기를 제안하고, 승인 시 원문으로 표기할 수 있다.
- frontmatter의 필드명과 태그는 영어로 통일한다.

# 버전 관리
- __MyNeuron/ 디렉토리는 git으로 버전 관리한다.
- 지식 추가/수정/정리 작업 후 적절한 단위로 commit한다.
- commit 메시지는 어떤 지식이 추가/수정/정리되었는지 알 수 있도록 작성한다.

# 이미지 관리
- 모든 이미지는 neuron/images/ 디렉토리에 통합 저장한다.
- 이미지 파일명은 관련 지식을 식별할 수 있도록 명명한다. (예: `transformer_attention_mechanism_01.png`)
- 이미지 파일명에 관련 지식 문서의 키워드를 포함하여, 특정 지식과 관련된 이미지들을 파일명 패턴으로 모아볼 수 있도록 한다. (예: `{문서키워드}_{설명}_{번호}.png`)
- 지식 문서에서 이미지를 참조할 때는 `![[이미지파일명]]` Obsidian 형식을 사용한다.
- 이미지를 생성하는 툴도 같이 생성한다. 이미지는 다이어그램을 직접 그릴 수도 있고, 웹서칭을 통해 적절한 그림을 가져올 수도 있고, 웹화면을 캡쳐하는 등 다양한 형식으로 얻을 수 있어야 한다.
- 이미지는 지식을 이해하는 데 도움이 되는 이미지여야하고, 지식을 이해하는 데 이미지가 있는 것이 좋다고 판단된다면 적극 활용해야 한다.
- 이미지를 발생시키는 것에 대해서는 활용될 수 있는 다양한 툴들을 만들어두고 적극 활용하며, 다른 방식으로 이미지를 얻는 것이 더 효율적이라고 판단되는 경우에는 또 다른 툴도 적극적으로 만들어도 된다. 
- 각종 툴들은 __MyNeuron/tools/에 저장한다.
